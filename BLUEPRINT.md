# 🗺️ 旅遊規劃藍圖 (Travel Planning Blueprint)

> **核心定位**：本文件定義了 AI 旅遊規劃師的規劃生命週期、資料合約與異動作業 SOP。  
> 任何新旅程的開啟與行程微調，皆必須嚴格遵循此規劃藍圖。  
> 程式碼實作與 HTML/JS 技術細節請參閱 [`FEATURES.md`](./FEATURES.md)。

## 1. 旅遊規劃的六大生命週期階段

作為專業規劃師，AI 在規劃與排定行程時，必須依照以下六個階段循序漸進，並以此作為決策與分析的主軸：

```
階段一：定錨與框架 ➔ 階段二：交通與住宿 ➔ 階段三：景點發想分類
       ↓                    ↓                    ↓
階段四：區域順序優化 ➔ 階段五：細節預約微調 ➔ 階段六：出發前夕檢核
```

### 1.1 階段一：定錨與框架（奠定基礎）
在深入查找景點前，必須先釐清旅程的「骨架」，決定整體的風格與限制：
* **成員與風格**：確認同行成員（獨旅、蜜月、好友群、或是長輩孝親）。成員屬性決定行程的步行速度、節奏（特種兵踩點 vs 悠閒度假）與主題屬性（購物、自然、網美打卡、文化深度）。
* **天數與預算**：扣除前後搭飛機，計算實際可玩天數。設定總預算與分配比例（機酒、餐飲、交通）。
* **鎖定大主題**：明確定義本趟旅程核心目的（例：自駕公路旅行、賞櫻、泡湯渡假）。

### 1.2 階段二：交通與住宿（鎖定大交通與大時程）
鎖定最吃預算且必須提早預訂的物流關鍵：
* **航班機票**：比價並確認抵達/離境時間。這會直接決定 Day 1 與最後一天的「實質可玩時數」。
* **點到點大交通**：確認交通模式（自駕、新幹線、國內線）。若自駕，需提前評估取還車地點、國際駕照/譯本。
* **住宿敲定**：根據交通樞紐與順路區域挑選高評價飯店，自駕旅客必須確認飯店是否好停車/有收費。

### 1.3 階段三：景點發想與分類（收集靈感）
收集所有想去的地方與美食，利用地圖分類標籤法（Color-Tagging）對資料進行結構化分類：
* **多管道收集**：從 Google Maps、社群平台與部落格發想景點、美食與購物清單。
* **地圖分類標籤法（ badge 設計基礎）**：
  * 🔴 **景點 (Sight)** ➔ 在網頁中對應 `sight` 類型，使用 **森林青綠 (`#3A5F43`)**
  * 🟢 **餐飲 (Food)** ➔ 在網頁中對應 `food` 類型，使用 **煙燻玫瑰粉 (`#C38B8B`)**
  * 🔵 **購物/預訂 (Buy/Booking)** ➔ 對應 `buy`/`booking`，使用 **灰調紫 (`#524675`) / 青黛藍 (`#3B7A87`)**
  * 🟡 **交通/住宿 (Transport/Hotel)** ➔ 對應 `transport`/`hotel`，使用 **石板灰 (`#5A6B7C`) / 紫羅蘭 (`#726297`)**

### 1.4 階段四：區域群組化與順序優化（核心排程）
將收集到的靈感進行視覺化排程，拒絕「折返跑」：
* **地理分區**：把距離相近的景點劃分在同一天。**一天安排 2-3 個主要景點 + 1-2 個備用點即可**，必須留白。
* **營業時間與公休日**：確實核對想去餐廳/美術館的公休日，避開週末與旺季的排隊尖峰。
* **線性/環狀原則**：路線規劃不論是市區一日遊或跨城自駕，皆須呈線性或單向環狀設計，嚴禁東奔西跑。

### 1.5 階段五：細節預約與微調（落實執行）
在整體骨架、住宿與景點排序確定後，規劃進入最關鍵的「執行細緻化」階段。AI 規劃師必須根據不同的事件類型，逐項對照並確認以下檢核要點，將細節融入網頁內容（如 tip 攻略或 notice 警告區塊），並將預約任務與截止時間自動寫入代辦清單，杜絕執行落差：

#### 🔴 景點 (Sight) 檢核清單
1. **確認營業時間與維修狀態**：核對官方網站與 Google Maps 營業時間，特別留意是否有臨時休館、局部整修（如古蹟圍封、遊樂設施保養）或季節性封閉。
2. **門票與搶票/排隊攻略**：確認門票售價，參考最新社群資訊（IG/Threads/Dcard 等）判斷是否需要提前搶票（如每日限額景點），並指引最划算、方便的購票管道（如 Klook、KKday、官網早鳥優惠）。
3. **參觀與美照重點**：參考社群最新討論與拍照打卡趨勢（IG/Threads/Dcard 等），提供最推薦的拍攝角度、不折返的參觀動線、必看展品或隱藏版觀景點。
4. **限定伴手禮與商品**：註明該景點獨有或近期在社群熱議的限定伴手禮、御守、扭蛋或聯名限定商品。
5. **自駕停車便利度**：若是自駕行程，必須查詢並提供最近且方便停放的停車場資訊（優先選擇免費或計次便宜的停車場），附上定位或 Mapcode。
6. **時間避峰指南**：針對熱門拍照與爆紅景點，給出避開人潮的黃金時段建議（如開門前 15 分鐘抵達、或清晨 7:00 包場拍照）。

#### 🟢 餐飲 (Food) 檢核清單
1. **預約方式與排隊對策**：確認是否可線上訂位（如 Tabelog、AutoReserve 或官網），若僅能電話預約，提醒利用信用卡秘書或飯店櫃檯代訂；若是現場排隊名店，參考社群（Threads/Dcard 等）回報的平均排隊時間，給出合理的提早排隊時間或避峰建議。
2. **必點招牌與隱藏吃法**：參考社群（IG/Threads/Dcard 等）美食推薦，註明必吃招牌菜色或點餐技巧（如特製沾醬、限定口味）。
3. **用餐規定與限制**：核對低消限制（如每人必點一杯飲料）、用餐限時、人數上限或年齡限制（如謝絕小學生以下）。
4. **公休避雷**：避開特定公休日（如海鮮市場與壽司店常於週三休市），並準備至少 1 個同區域的免排隊/快速用餐備案 (Plan B)。
5. **附近停車位**：確認餐廳是否有自用車位，或提供最近的付費停車場資訊。

#### 🟡 交通/自駕 (Transport) 檢核清單
1. **行車路線與山路霧季緩衝**：評估路況（如箱根、伊豆等山區窄路或多霧路段），自駕行車時間應在 Google Maps 的基礎上額外加上 15-20% 的安全緩衝。
2. **還車滿油與加油站定位**：在還車點附近尋找並標記「距離最近、好進出」的加油站，註明適用油品（如 Regular 紅色油槍），並提醒保留收據供店員查驗。
3. **大眾運輸轉乘找路緩衝**：若搭乘電車且需要在新宿、東京等複雜大站轉乘，必須預留至少 15-20 分鐘的步行與找路時間。
4. **票券划算度評估**：針對全程自駕者，明示「切勿購買大眾運輸周遊券（如箱根 Free Pass）」，避免重疊消費。

#### 🟡 住宿 (Hotel) 檢核清單
1. **入住窗口與辦理方式**：溫泉旅館需留意最晚入住時間（避免錯過晚餐供應）；無人自助公寓（如樂天 STAY）需註明平板辦理入住的步驟與 QR Code 領取方式。
2. **車位預留與車輛規格限制**：確認飯店停車位收費標準、是否需要提前預約，以及立體停車塔的車高/車寬限額。
3. **行李寄存與寄送代收**：確認退房後行李寄放規定，以及是否能協助代收/寄送大件行李（黑貓宅急便等）。

#### 🔵 預訂/票券 (Booking) 檢核清單
1. **憑證備份與離線存取**：所有 PDF 憑證、訂單編號與 QR Code 需上傳至雲端，並下載離線檔案到手機，避免現場網路訊號不佳。
2. **實體票券兌換動線**：詳細指引抵達現場後的取票機台、櫃檯位置與操作步驟（如 Skyliner 自助機台 QR Code 兌票步驟）。

### 1.6 階段六：出發前夕檢核表（Final Check）
出發前 1-2 週進行最終確認，檢核表應包含：
* **電子備份**：機票、飯店確認單、租車憑證存入手機，下載離線地圖。
* **實體證件**：護照效期（6個月以上）、國際駕照/譯本。
* **金流準備**：外幣現金、海外高回饋信用卡。

## 2. 新專案啟動標準流程

當使用者發起一個新專案時，規劃師 AI 必須依循以下程序：

### 2.1 啟動必問問卷 (對應階段一、二)
在建立任何行程網頁前，先請使用者提供以下 9 大資訊：
```
1. 出發地 & 目的地：
2. 旅遊天數：
3. 旅遊日期：
4. 同行成員：(人數與組成，如：2長輩4中青)
5. 交通資訊：(全程自駕 / 部分租車 / 全程大眾運輸)
6. 路線資訊：(例如：福岡進出順時針環島)
7. 住宿資訊：(已訂妥或待推薦)
8. 必去景點/活動：
9. 必吃美食：
```

### 2.2 提案回覆標準範本 (對應階段三、四、五)
AI 收到問卷後，必須以下列架構回覆提案：
> #### 🗂️ 專案初始化提案
> **1. 旅程基本盤**：[天數、成員、日期摘要]  
> **2. 規劃師觀點與亮點推薦**：
> * *季節限定*：我查到您去的時間剛好有 [活動]，已幫您融入 Day X！
> * *成員友善*：因為有 [長輩/小孩]，Day Y 行程改為 [替代方案]。
> * *解鎖體驗*：建議 Day Z 升級為 [一泊二食/划船]，預算約多 [金額]。  
> **3. 路線優化**：原 A➔B 較繞，改為 A➔C➔B。  
> **4. 備案與待辦**：雨天 Plan B 為 [備案]，[景點X] 門票需在 [日期] 搶購，已幫您加入代辦清單。

## 3. 行程景點異動 SOP

當行程網頁建立後，使用者要求「新增/修改/調整景點」時，AI 必須嚴格執行以下 SOP：

```
【SOP 流程】
Phase 1: 問確認 (階段一) ➔ 放哪天？幾點？停多久？取代或插入？
Phase 2: 查資料 (階段三) ➔ 營業時間、公休、票價、停車場狀況、隱形時間成本。
Phase 3: 衝檢   (階段四) ➔ 前後交通衝突？最後入場？長輩體力？Plan B 犧牲哪項？
Phase 4: 寫入   (階段五) ➔ 建立 events 物件，有預訂需求則自動加 notesChecklistData
Phase 5: 部署   (階段六) ➔ git add . ; git commit -m "feat(itinerary): ..."; git push
```

## 4. 旅程清單設計與生成原則

本節定義各項旅程輔助清單的「規劃與生成邏輯」，具體的 JavaScript JSON 資料結構與合約規格請參閱 [`FEATURES.md` 第 4 節](./FEATURES.md#4-資料結構與合約-data-structures--contracts)。

### 4.1 購物清單設計原則
* **使用者自訂主導**：網頁初始化時，`shoppingList` 的 items 一律為空 `[]`。AI 不得擅自預填項目，僅提供基礎類別框架，交由使用者自行新增。

### 4.2 行李清單打包原則
* **環境與成員適配**：AI 必須根據目的地地形、季節溫度及同行成員組成（如長輩護膝、防滑徒步鞋），主動為本趟旅程推薦專屬打包行李清單，拒絕死板範本。

### 4.3 注意事項代辦清單原則
* **行程事件強關聯**：代辦清單的項目必須動態與行程中的特定景點（如高千穗划船搶票）或交通方式（如自駕譯本、大車空間預訂）完全掛鉤，確保每項代辦都有具體的行程依據與截止時程。

## 5. 視覺設計與美學規範

為維持旅遊雜誌般的精緻排版，配色與區塊必須遵守以下原則：

### 5.1 莫蘭迪低飽和度配色 (Morandi Tint Palette)
卡片背景採用 **10% 主題淡色背景 (Tint Background)**，僅在 Icon 和標題使用主色：

| 類型 | 主色 (Text/Icon) | 背景色 (Tint bg) | 邊框色 |
| :--- | :--- | :--- | :--- |
| **`sight` 景點** | `#3A5F43` (森林綠) | `#F2F6F3` | `#E2ECE5` |
| **`transport` 交通** | `#5A6B7C` (石板灰) | `#F4F6F8` | `#E5EAEF` |
| **`food` 餐飲** | `#C38B8B` (玫瑰粉) | `#FAF1F1` | `#F5E3E3` |
| **`buy` 購物** | `#524675` (灰調紫) | `#F5F3F9` | `#E7E3F0` |
| **`flight` 航班** | `#A863A0` (洋紅紫) | `#F8F2F9` | `#EFE2F1` |
| **`hotel` 住宿** | `#726297` (紫羅蘭) | `#F4F3F7` | `#E6E3ED` |
| **`booking` 預訂** | `#3B7A87` (青黛藍) | `#EEF5F6` | `#DCECEF` |
| **`story` 故事** | `#57534E` (暖石灰) | `#FAFAF9` | `#EBEAE6` |

### 5.2 ⚠️ 警告區塊收斂 (Clean Notice UI)
* 不論是 `danger` 或 `warning` 類型的 Notice 區塊，**底色一律收斂為淡米色 (`#FCF8F2`)**，去除大片紅色或黃色底色。
* 僅透過文字顏色（特別重要：紅字 `#9B2C2C` / 一般注意：黃字 `#6B512C`）與驚嘆號顏色做細緻的級別區隔。
